## 03-6: 新增少数派 RSS 订阅源

### 当前状态

- 目前仅有 1 个订阅源：老布的AI知识库
- 用户希望添加少数派（sspai.com）作为第二个信息源
- 少数派 RSS feed 地址：`https://sspai.com/feed`
- 少数派内容领域：数字生活、AI、硬件、Web 技术、影视、效率工具、社区活动等

### 少数派内容分析

从 feed 近期文章看，少数派覆盖的主要领域：

| 领域 | 示例文章 | 与现有分类匹配 |
|------|----------|---------------|
| 数字遗产/隐私 | 「数字遗产」保护 | ✅ 隐私与数字生活 |
| AI 硬件 | 千问 AI 眼镜 G1 | ✅ AI与智能体 / 硬件与设备 |
| 影视娱乐 | 本周看什么、超级少女预告 | ✅ 影视与娱乐 |
| Web 技术 | 网页变应用程序的历史 | ✅ 开发与Web技术 |
| 手机硬件 | vivo X300 Ultra | ✅ 硬件与设备 |
| AI 模型 | Gemma 4、GLM-5V-Turbo（派早报） | ✅ AI与智能体 |
| 社区活动 | 2050 大会 | ✅ 社区与活动 |
| 公司史 | 苹果 50 年 | ✅ 商业与公司史 |
| 日报/晚报 | 派早报、派晚报 | ⚠️ 需新增"资讯速递"分类 |

### 实施内容

1. **云端操作**：通过 lumen CLI 添加少数派订阅源
   ```bash
   # SSH 登录腾讯云后执行
   cd /root/tencent_rsync_mac/RSS_article
   ./lumen feed add https://sspai.com/feed
   ./lumen fetch
   ```
2. **验证 feeds.db**：确认少数派文章成功写入，检查 feed_id 和文章数量
3. **多源推送测试**：手动触发 Agent，确认两个源各自独立生成简报
4. **分类规则适配**：少数派有"派早报/派晚报"类聚合文章，需新增"资讯速递"分类
   - 新增关键词：`早报`, `晚报`, `日报`, `周报`, `速递`, `资讯`, `资讯聚合`

### 验收标准

- [ ] lumen 成功添加少数派订阅源
- [ ] `lumen fetch` 成功抓取少数派文章
- [ ] feeds.db 中可见少数派的文章数据
- [ ] 手动触发 Agent 后，飞书群收到 2 条独立简报（老布 + 少数派）
- [ ] 少数派文章的主题分类合理（≤10% 未分类）
- [ ] "派早报/派晚报"正确归类为"资讯速递"

### 风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 少数派 feed 有反爬机制 | 无法抓取 | 低 | lumen 支持 user-agent 配置；或使用 RSSHub 中转 |

## 03-1: Gateway systemd 服务配置

### 当前状态

- OpenClaw Gateway 通过非 systemd 方式运行
- `gateway restart` 命令不生效
- 服务崩溃后需手动重启

### 目标

| 项目 | 目标值 |
|------|--------|
| 自动重启 | 进程崩溃后 5 秒内自动拉起 |
| 日志管理 | journalctl 可查日志，自动轮转 |
| 开机自启 | 服务器重启后 Gateway 自动运行 |

### 实施内容

1. 创建 systemd service 文件 `/etc/systemd/system/openclaw-gateway.service`
2. 配置 Restart=on-failure、RestartSec=5
3. 配置 WorkingDirectory、ExecStart、User=root
4. 启用并启动服务：`systemctl enable --now openclaw-gateway`
5. 验证：手动 kill 进程后观察自动恢复

### 上游产物

- `architecture.md` — 部署架构、错误处理策略

## 03-2: lumen fetch 健康检查

### 当前状态

- Agent 依赖 lumen fetch 写入 feeds.db，但 Agent 不检查 lumen 是否正常运行
- 若 lumen 长时间未 fetch，Agent 查询结果为空，用户无感知

### 目标

- Agent 执行时检查 lumen 最近一次 fetch 时间
- 超过 48 小时未 fetch 则在简报中附带告警信息

### 实施内容

1. 在 SOUL.md Step 1 后新增健康检查步骤
2. 检查 feeds 表中最新的 article published_at 或 feeds 表的 last_fetch_at
3. 若最近 fetch 时间超过 48h，在输出中追加告警：
   ```
   ⚠️ 注意：lumen 已超过 48 小时未抓取新文章，请检查 lumen cron 是否正常。
   ```
4. 告警不影响正常推送逻辑

### 上游产物

- `architecture.md` — 错误处理策略
- `ux-flow.md` — 异常场景交互

## 03-3: 异常场景测试

### 测试场景

| # | 场景 | 操作 | 预期结果 | 优先级 |
|---|------|------|----------|--------|
| T1 | feeds.db 不存在 | 临时重命名 feeds.db | Agent 输出错误日志，不推送，不崩溃 | P0 |
| T2 | digest.db 损坏 | 写入非法数据到 digest.db | Agent 自动重建表（CREATE IF NOT EXISTS），恢复正常 | P1 |
| T3 | feeds.db WAL 锁 | 模拟并发写入 | 只读查询正常返回，不阻塞 | P2 |
| T4 | 空结果（无新文章） | 确认 digest.db 已记录所有文章 | 输出"今日暂无新文章"，不推送 | P0 |
| T5 | lumen 长时间未 fetch | 检查 published_at 均为旧数据 | 附带告警信息（03-2 实现后测试） | P1 |

### 验收标准

- [ ] T1: feeds.db 缺失时 Agent 优雅终止
- [ ] T2: digest.db 损坏时自动恢复
- [ ] T4: 空结果正确处理
- [ ] 所有测试结果记录到 docs/test/
