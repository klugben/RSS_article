# 问题定义

> 项目: RSS_article
> 状态: 已确认

## 核心问题

信息过载环境下，个人用户每天面对多个 RSS 订阅源产出的文章，缺乏高效的方式快速获取当日关键信息。

## 角色

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 信息消费者 | 关注 AI/科技领域的个人用户 | 每天花 < 2 分钟了解当日最重要的 RSS 文章内容 |
| 订阅源管理者 | 维护 RSS 抓取系统的用户 | 订阅源增删改后自动生效，无需修改推送逻辑 |

## 核心目标

1. **自动化**: 每日定时抓取、聚合、摘要、推送，无需人工介入
2. **精简性**: 简报总字数 ≤ 400 字，适配飞书群消息阅读体验
3. **可靠性**: 有新文章就推送（哪怕 1 篇），无新文章则静默

## 问题边界

### 做什么

- 从 SQLite 数据库读取最近 24 小时新增文章
- 按主题关键词规则分类
- 生成结构化摘要并推送到飞书群组
- 通过 OpenClaw Cron 实现每日 8:00 定时触发

### 不做什么

- 不做全文推送
- 不做个性化推荐（所有群成员看到相同简报）
- 不做文章收藏/标记/历史检索
- 不做多渠道推送（仅飞书）
- 不做图片/富媒体处理

## 关键约束

1. 飞书消息阅读体验限制简报在 400 字以内
2. 云端服务器部分图片源返回 403，简报不依赖图片
3. SQLite 可能被 lumen fetch 和导出脚本同时访问，查询必须只读
4. 云端服务器时区需确认（UTC → 北京时间 UTC+8 转换）

## 假设

- 每日新增文章通常 0-5 篇（当前仅一个订阅源）
- OpenClaw Gateway 正常运行时 Cron 才能触发
- 飞书群组已添加 OpenClaw Bot

## 已决问题

1. **已推送去重** — 需要。记录已推送文章 ID，Cron 多次触发或 24h 窗口重叠时不重复推送同一篇文章。
2. **多订阅源扩展** — 每个订阅源独立生成一份日报。多源 = 多条推送消息，互不干扰，各自遵守 400 字限制。
3. **手动触发** — 支持。用户可通过 `openclaw agents run` 或直接调用 Agent 手动触发推送，不受 Cron 限制。

## 不做什么（补充）

- 不做多源合并推送（每源独立日报）
- 不做推送历史管理 UI（通过 DB 记录实现去重即可）
