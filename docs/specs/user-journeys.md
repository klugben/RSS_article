# 用户旅程

> 项目: RSS_article
> 状态: 已确认

## 旅程 1：日常信息获取（核心旅程）

```
早晨 8:00 ──▶ 飞书群收到简报消息
                    │
                    ▼
              扫一眼标题区（< 5 秒）
                    │
              ┌─────┴─────┐
              │ 有感兴趣   │ 全部不感兴趣
              │ 的内容？   │
              ▼           ▼
         点击原文链接    关闭消息
         深度阅读       （任务完成）
              │
              ▼
         获得新知/见解
         （价值实现）
```

### 场景变体

**场景 A：文章少（1-3 篇）**
- 简报展示每篇文章的一句话摘要 + 原文链接
- 用户逐条浏览，选择性点击

**场景 B：文章多（>3 篇）**
- 简报展示 2-3 句要点概括 + 全部文章链接列表
- 用户先看要点，再决定哪些值得深读

**场景 C：无新文章**
- 不推送任何消息
- 用户不被打扰

## 旅程 2：新订阅源添加（管理旅程）

```
用户添加新 RSS 订阅源到 lumen
                    │
                    ▼
         lumen fetch 抓取新源文章
                    │
                    ▼
         文章写入 feeds.db（已有 schema）
                    │
                    ▼
         下次 Cron 触发时自动包含新源文章
         （无需修改 Agent 逻辑）
```

### 关键假设
- lumen 的 articles 表结构兼容不同订阅源
- 主题分类关键词规则覆盖新源的内容领域

## 旅程 3：异常处理

**异常 1：Cron 未触发（Gateway 异常）**
```
用户发现当天没收到简报
        │
        ▼
  手动 SSH 到服务器
        │
        ▼
  检查 OpenClaw Gateway 状态
        │
        ▼
  重启 Gateway / 手动触发推送
```

**异常 2：摘要质量不满意**
```
用户发现简报内容不准确
        │
        ▼
  调整 SOUL.md 中的行为规则
        │
        ▼
  重新部署到 Agent workspace
```

## 旅程度量

| 旅程 | 成功指标 | 失败信号 |
|------|---------|---------|
| 日常信息获取 | 用户在 < 2 分钟内完成浏览 | 用户仍需打开 lumen 逐篇阅读 |
| 新订阅源添加 | 新源文章次日自动出现在简报 | 需要手动修改代码 |
| 异常处理 | 异常在 1 次操作内恢复 | 需要多次调试 |
