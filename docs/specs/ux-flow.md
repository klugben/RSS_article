# UX 交互流

> 项目: RSS_article
> 状态: 已确认

## 设计说明

本项目无 Web UI，用户交互发生在两个层面：
1. **被动接收**：飞书群消息简报
2. **主动操作**：手动触发推送 / 管理 Agent 配置

## 一、核心交互流：每日简报接收

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Cron 触发   │────▶│  Agent 处理  │────▶│  飞书推送    │
│  每日 8:00   │     │  < 30 秒    │     │  群消息      │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                                │
                                                ▼
                                         ┌─────────────┐
                                         │  用户打开    │
                                         │  飞书群消息  │
                                         └──────┬──────┘
                                                │
                                    ┌───────────┴───────────┐
                                    │  扫一眼标题区（<5秒）  │
                                    └───────────┬───────────┘
                                                │
                                    ┌───────────┴───────────┐
                                    │ 有感兴趣的内容？       │
                                    └───┬───────────────┬───┘
                                       Yes              No
                                        │               │
                                        ▼               ▼
                                ┌──────────────┐  ┌──────────┐
                                │ 点击原文链接  │  │ 关闭消息  │
                                │ 深度阅读     │  │ 完成     │
                                └──────────────┘  └──────────┘
```

## 二、交互场景

### 场景 1：文章少（≤3 篇）

**用户看到**：每篇文章独立展示，一句话摘要 + 原文链接

**交互行为**：
1. 扫标题区 → 看到各主题分类
2. 逐条浏览摘要
3. 选择性点击原文链接

**满意条件**：2 分钟内完成浏览，找到值得深读的文章

### 场景 2：文章多（>3 篇）

**用户看到**：顶部要点概括（2-3 句），下方全部文章链接列表

**交互行为**：
1. 先读要点概括（10 秒）
2. 扫链接列表中的标题
3. 选择性点击原文链接

**满意条件**：1 分钟内判断哪些值得深读

### 场景 3：无新文章

**用户看到**：无消息（静默）

**交互行为**：无

**满意条件**：不被打扰

### 场景 4：多订阅源

**用户看到**：每源独立消息，各消息遵守 400 字限制

**交互行为**：逐条浏览，每条消息独立消费

**满意条件**：消息之间不混淆，每源内容清晰

## 三、管理交互流

### 3.1 手动触发推送

```
用户在终端执行
        │
        ▼
openclaw agents run RSS_article
        │
        ▼
Agent 执行完整流程（查询→去重→分类→摘要→推送）
        │
        ▼
终端输出执行结果（推送了 N 篇文章 / 无新文章）
```

**触发场景**：
- Cron 未触发（Gateway 异常）后补推
- 测试 Agent 配置变更
- 临时需要重新生成简报

### 3.2 配置变更

```
用户修改 SOUL.md / USER.md
        │
        ▼
同步到云端 Agent workspace
        │
        ▼
手动触发验证
        │
        ▼
确认简报格式/内容符合预期
```

### 3.3 新增订阅源

```
用户通过 lumen 添加新 RSS 源
        │
        ▼
lumen fetch 抓取文章 → 写入 feeds.db
        │
        ▼
下次 Cron 触发时自动包含新源文章
（无需修改 Agent 逻辑）
```

## 四、消息格式规范

### 格式原则

| 原则 | 说明 |
|------|------|
| 扁平结构 | 不使用多级标题，最多 2 级 |
| 即时消费 | 无需展开/折叠，所有信息一屏可见 |
| 链接可达 | 每篇文章都有原文链接 |
| 字数严格 | 总字数 ≤ 400（含标题和分类标签） |
| 无 emoji | 标题 📰 除外 |

### 信息层次

```
第 1 层：简报标题（日期 + 来源）
    │
第 2 层：主题分类（## 标题）
    │
第 3 层：文章摘要 + 链接
```

## 五、异常场景交互

| 异常 | 用户感知 | 用户操作 |
|------|---------|---------|
| Cron 未触发 | 收不到简报 | SSH 检查 Gateway，手动触发 |
| DB 查询失败 | 收不到简报 | 检查 DB_PATH，检查 lumen 状态 |
| 推送失败 | 收不到简报 | 检查 chat_id 配置，检查飞书 Bot 状态 |
| 摘要超字数 | 消息被截断 | 修改 SOUL.md 压缩规则 |
